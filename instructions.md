# tg-movie-bot — инструкция для будущих сессий

Этот файл — единственный источник правды о том, **почему** код устроен именно так.
Если код и этот файл расходятся — доверяй коду, но сначала спроси себя, не забыли ли
здесь что-то обновить после последней правки.

## 1. Архитектура

**Стек:**
- Node.js, ES Modules (`"type": "module"` в `package.json`)
- [`telegraf`](https://telegraf.js.org/) 4+ — Telegram Bot API
- `better-sqlite3` — локальная синхронная SQLite-база (`database.db`, в `.gitignore`)
- `dotenv` — переменные окружения из `.env`
- Встроенный `fetch` (Node 18+) — HTTP-запросы к TMDB, без дополнительных пакетов

**Файлы в корне проекта:**
- `bot.js` — вся логика бота: конфигурация категорий/жанров, обращения к TMDB,
  форматирование карточек, все `bot.action`/`bot.hears`/`bot.on` обработчики.
- `db.js` — схема SQLite + мягкие миграции (`ALTER TABLE ... ADD COLUMN`, без
  разрушительных `DROP`). Экспортирует готовый `db` инстанс.
- `.env` — `TELEGRAM_BOT_TOKEN`, `TMDB_API_KEY`, `REZKA_BASE_URL`.
- `package.json` — `"start": "node bot.js"`.

**Важно про источник данных:** каталог фильмов **не хранится статично**. Раньше
(на ранних итерациях) в `db.js` был захардкожен список из 10–38 фильмов — от этого
полностью отказались. Сейчас всё подбирается "на лету" через живой TMDB API
(`/discover/movie`, `/discover/tv`, `/search/multi`). Таблица `movies` в SQLite —
это **кэш** уже показанных пользователю карточек (нужен, чтобы `watched`/`watchlist`
и повторный показ в "Моём списке" работали мгновенно, без нового похода в TMDB), а
не источник истины.

**Схема БД (`db.js`):**
```
users (
  id TEXT PRIMARY KEY,        -- Telegram user id
  state TEXT,                 -- null | 'awaiting_mood' | 'awaiting_search'
  temp_type TEXT,             -- категория, выбранная на Шаге 1, до завершения Шага 2
  last_category TEXT,         -- категория последней успешной выдачи (для "Другой вариант")
  last_genre TEXT,            -- жанр/вайб последней успешной выдачи
  last_movie_id TEXT,         -- id последней показанной карточки (чтобы не повторить её же)
  -- temp_country, temp_genre, temp_criteria, temp_year, last_type, last_year —
  -- НАСЛЕДИЕ прошлых версий опросника (5 шагов, потом 3), в коде больше не
  -- используются. Не удалены из схемы намеренно — миграции в этом проекте
  -- только аддитивные (ALTER TABLE ADD COLUMN), никогда не дропаем колонки,
  -- чтобы не терять данные при апдейте на середине разработки.
)

movies (
  id TEXT PRIMARY KEY,        -- составной: `${media_type}_${tmdb_id}`, напр. "movie_27205"
  tmdb_id INTEGER, media_type TEXT,  -- 'movie' | 'tv'
  title, original_title, year, vote_average, overview, poster_path
)

watched (user_id TEXT, movie_id TEXT, PRIMARY KEY(user_id, movie_id))
watchlist (user_id TEXT, movie_id TEXT, PRIMARY KEY(user_id, movie_id))
```

**Поток пользователя:**
1. `/start` или кнопка Reply-меню "🤖 Подобрать кино" → inline-кнопки 5 категорий (Шаг 1).
2. Выбор категории → inline-кнопки жанров/вайбов, свои для каждой категории (Шаг 2).
3. Выбор жанра → бот **сразу** делает запрос к TMDB и присылает **одну** карточку
   (постер + название рус./ориг. + год + рейтинг TMDB + описание рус. + кнопка
   "🍿 Смотреть" на поиск HDRezka + "✅ Смотрел" + "📌 Хочу посмотреть" +
   "🔄 Другой вариант").
4. Опрос — **ровно 2 шага** (категория → жанр). Многошаговых версий с выбором
   типа/страны/эпохи/критерия рейтинга (было на ранних итерациях, вплоть до 5 шагов)
   **больше нет** — от них сознательно отказались как от "слишком запутанных".

Остальные функции реального меню: "🔍 Поиск по названию" (живой поиск через
`/search/multi`, не завязан на категории/жанры) и "📌 Мой список" (`watchlist`,
читается из локального кэша `movies`, без обращения к TMDB).

## 2. Правила фильтрации по категориям

Все 5 категорий и их наборы жанров живут в `bot.js` в константах `CATEGORIES`,
`STANDARD_MOODS`, `TV_MOODS`, `CARTOON_MOODS`, `ANIME_MOODS`.

Общее для всех категорий: `language=ru-RU`, `sort_by=popularity.desc`, обязательно
постер (`poster_path` не пустой — TMDB не даёт такого параметра запроса, поэтому
фильтруется на нашей стороне после ответа).

### 🎬 Популярные фильмы / 🍿 Свежие новинки
- `movie`, `without_genres=16` (жёсткое правило: без этого в выдачу попадали
  мультфильмы вроде "Зверополиса" — баг, который явно чинили).
- Популярные: `vote_count.gte=1000`, `vote_average.gte=6.5`.
- Новинки: `vote_count.gte=150`, `primary_release_date.gte=2025-01-01`.
- Жанры Шага 2 — `STANDARD_MOODS`: Комедия(35), Экшен/Фантастика(28,878),
  Триллер/Ужасы(53,27), Драма(18), Любой жанр. Здесь комбинации через
  запятую **разрешены** (TMDB трактует запятую как AND).

### 📺 Крутые сериалы
- `tv`, `without_genres=16`.
- **Смягчённые пороги** (специально ниже, чем у фильмов): `vote_count.gte=50`,
  `vote_average.gte=6.5`.
- **ЖЁСТКОЕ ПРАВИЛО:** для сериалов жанры **никогда** не объединяются через
  запятую — каждая кнопка Шага 2 (`TV_MOODS`) шлёт ровно один ID жанра:
  Фантастика/Фентези(**10765**), Боевик/Приключения(**10759**), Комедия(35),
  Детектив/Криминал(80), Драма(18), Любой сериал. Обрати внимание: 10765/10759 —
  это TV-специфичные ID жанров TMDB (Sci-Fi & Fantasy / Action & Adventure),
  **не те же самые**, что 28/878 для фильмов — TMDB использует разную таксономию
  для movie и tv.

### 👶 Мультфильмы
- `movie` + `tv` одновременно (пользователю всё равно, вышло это фильмом или
  сериалом).
- `with_genres=16`, `vote_count.gte=300`, `vote_average.gte=6.5`.
- **Исключение аниме:** `without_original_language=ja`. ⚠️ Это **не настоящий
  параметр TMDB API** — если отправить его как query-параметр, TMDB его молча
  проигнорирует (проверено живым запросом). Поэтому исключение реализовано
  **клиентской фильтрацией** уже полученных результатов:
  `raw.filter(r => r.original_language !== 'ja')` (см. `discoverCandidates`,
  поле `category.excludeOriginalLanguage`).
- Интерфейс Шага 2 — **полностью на русском**: "😂 Веселые и семейные",
  "🔮 Сказки и Магия", "🚀 Приключения и супергерои", "🎲 Любой мультик".

### ⛩️ Топовое Аниме
- `movie` + `tv` одновременно.
- `with_genres=16`, `with_original_language=ja` — а вот это **настоящий,
  рабочий параметр TMDB** (в отличие от `without_original_language`), передаётся
  прямо в query.
- `vote_count.gte=150`, `vote_average.gte=7.0`.
- Жанры Шага 2 на русском: "⚔️ Эпический экшен (Сёнен)", "🎒 Повседневность и
  комедия", "😢 Глубокие шедевры (Для души)", "👻 Темное фэнтези / Триллер",
  "🎲 Любое аниме".

### Технический трюк: жанры через "ИЛИ" для Мультфильмов/Аниме
Часть пунктов Шага 2 для Мультфильмов/Аниме комбинирует жанры через "ИЛИ"
(например, "Веселые и семейные" = жанр 35 **или** 10751). TMDB **не умеет**
мешать AND и OR в одном параметре `with_genres` (запятая всегда AND для всей
строки). Поэтому такие пункты хранятся в конфиге как массив `genreGroups` из
нескольких альтернатив (например `['35', '10751']`), и `discoverCandidates`
делает **отдельный запрос на каждую группу** (`16,35` и `16,10751` отдельно),
затем объединяет и дедуплицирует результаты по `id`. Это и есть OR — но на
уровне нескольких запросов, а не одного параметра.

Известный побочный эффект: для некоторых комбинаций (например, Аниме +
"Сёнен" = 28/12 для `tv`) TMDB отдаёт 0 результатов **только для tv** — потому
что у TMDB разная таксономия жанров для фильмов и сериалов. Так как категория
всегда ищет по `movie` **и** `tv` одновременно, пустая tv-часть не страшна:
результаты по `movie` всё равно есть. Это не баг, а особенность данных TMDB
(проверено и задокументировано в истории правок).

## 3. Логика "🔄 Другой вариант" (replace_movie)

⚠️ Важное уточнение: реализация **не** использует in-memory `userStates` — вся
подбор-сессия персистентна в SQLite (таблица `users`), переживает перезапуск
бота. Опрос — 2 шага, поэтому "фильтров" для сохранения ровно **два**:
категория и жанр (не пять — упоминание "5 фильтров" в задаче отражает более
раннюю версию опросника, которой в текущем коде уже нет).

Как это устроено:
1. После успешного подбора карточки (`mood:...` обработчик) бот пишет в БД:
   `last_category` (ключ категории), `last_genre` (ключ жанра/вайба),
   `last_movie_id` (id показанного фильма) — это происходит **уже после**
   сброса `state`/`temp_type` (которые нужны только во время самого опроса),
   так что "Другой вариант" продолжает работать даже после того, как обычный
   стейт опроса очищен.
2. Кнопка "🔄 Другой вариант" на карточке имеет **фиксированный** `callback_data:
   'replace_movie'` (без id фильма в data) — так решили сознательно. Это значит,
   что если в чате осталась старая карточка из давней категории и нажать на её
   кнопку, замена произойдёт по **последней** сохранённой категории/жанру, а не
   по той, с которой была создана именно эта карточка. Это принятый компромисс,
   а не баг.
3. Обработчик `bot.action('replace_movie', ...)`: читает `last_category`/
   `last_genre` из БД → находит соответствующие `category`/`mood` в конфиге →
   исключает и весь `watched`, и `last_movie_id` (чтобы не показать то же самое
   ещё раз) → делает **точно такой же** запрос к TMDB, что и при исходном
   подборе (`discoverCandidates(category, mood, excludeIds, page)`) → берёт
   случайный из непросмотренных → **удаляет старое сообщение и отправляет новое**
   (не редактирует медиа: Telegram не даёт надёжно поменять фото на текст и
   обратно через `editMessageMedia`, поэтому выбран более простой и надёжный
   путь "удалить + переслать").
4. Если `last_category`/`last_genre` не нашлись (очень старая сессия до этой
   фичи) — бот честно пишет "Не нашёл сохранённые фильтры" и просит начать
   заново, вместо тихого фолбэка на дефолтные настройки.

## 4. Языковой стандарт

Весь пользовательский интерфейс (тексты кнопок, названия категорий/жанров,
основные сообщения меню) — **на русском языке**. Ни украинского, ни смешения
языков в UI быть не должно.

украинский текст для двух сообщений об ошибках):
```js
const ERROR_MSG = 'Ой, виникла помилка при пошуку фільмів. Спробуйте ще раз!';
// и в обработчике Шага 2:
```

Если действующее правило — "всё строго на русском без исключений", эти две

собственному более раннему явному запросу пользователя — стоит уточнить перед
правкой, а не менять молча).

## 5. История ключевых решений (коротко, чтобы не наступать на те же грабли)

- Отказались от статичного каталога фильмов в БД → перешли на живой TMDB API.
- Callback-обработчики: **всегда** `await ctx.answerCbQuery().catch(() => {})`
  первой строкой — иначе Telegram кидает `400: query is too old`, и без
  `try/catch` вокруг это роняло весь процесс Node (было дважды: до общего
  `bot.catch()` и до этого правила).
- Промежуточные `ctx.editMessageText(...)` в процессе опроса всегда с
  `.catch(() => {})` — иначе `400: message is not modified` (при повторном клике
  с тем же текстом) прерывает обработку без ответа пользователю ("зависание").
- Карточка может быть фото (с постером) или текстовым сообщением (без него) —
  редактировать её нужно `editMessageCaption` или `editMessageText`
  соответственно, проверяя `ctx.callbackQuery.message.photo` (`editCardMessage`).
- Опрос эволюционировал: 5 шагов (тип/жанр/страна/критерий/эпоха) → 3 шага →
  сейчас 2 шага (категория → жанр). Категории "Мультфильмы" и "Аниме" один раз
  объединяли в одну кнопку, потом снова разделили — сейчас они строго раздельные,
  каждая ведёт на свой Шаг 2.
- `without_original_language` не существует как параметр TMDB — TMDB его молча
  игнорирует. Единственный способ исключить язык — фильтровать ответ на своей
  стороне. `with_original_language`, наоборот, реальный и рабочий параметр.
- Жанры через запятую в `with_genres` — это AND у TMDB, не OR (легко перепутать
  интуитивно). Для настоящего OR нужно делать несколько запросов и объединять
  результаты (см. раздел 2, "Технический трюк").
- Миграции БД — только аддитивные (`ALTER TABLE ADD COLUMN`), никогда не дропаем
  существующие колонки/таблицы, чтобы не терять `watched`/`watchlist` при
  очередной переделке опросника.
